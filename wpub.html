要約

この仕様の主な目的は、ウェブ出版物の制作における要件を定義することである。その際、ウェブの経路が非常に望ましいが、必ずしも交換又は消費の主要な方法ではないので、例えば、EPUB やオーディオブックなど、パッケージ化された出版物フォーマットを作成するためのフレームワークも定義する。

このドキュメントのステータス

このセクションは、このドキュメントの発行時のステータスについて説明する。他のドキュメントが、このドキュメントに優先する場合がある。現在の W3C の出版物のリストとこの技術レポートの最新版は、https://www.w3.org/TR/ の W3C 技術レポートのインデックスで見つけることができる。

ウェブ出版物は実用的なビジネス事例が不足しており、技術を実装するコミットメントが不足している結果から、Publishing Working Group は覚書としてこのドキュメントを発行し、別の仕様としてマニフェスト フォーマットを開発するなど、他の関心領域に焦点を当てることを選択した。

このドキュメントは、その発行時点でまだ進行中の作業であった。そのため、ウェブ出版物を作成したり、それらのリーダーを実装したりする場合は、このドキュメントで概説されているアプローチと提案を十分に注意して読まれたい。将来その開発を再開するための関心が生じた場合に備えて、作品をアーカイブし、インキュベーションを可能にするために発行されている。

特に、仕様の次の側面は、発行時に活発に議論されていたものであり、不完全であるか、さらなるレビューが必要であると見なされている。

  • 正規の識別子の性質、
  • アドレスが特定のリソースを指定するか、ディレクトリのみを参照するかどうかに関わりなく（つまり、提供されるデフォルトのドキュメントに依存する）、
  • プライマリ エントリ ページからマニフェストへのリンク及び出版物リソースからプライマリ エントリ ページへのリンクの違い。そして、
  • マニフェストの処理、特に（例えば、タイトルや言語情報など）プライマリ エントリ ページからの情報の継承。

同様に、次の領域のレビューは、仕様が安定したときにのみ、実施されることを意図している。

  • 例えば、オリジン間リソース共有に関する問題やウェブ出版物リソースの厳格なオリジンなどのセキュリティ、
  • 保存されたコンテンツのメンテナンスとユーザーが識別可能な情報に関するプライバシー。そして、
  • 機能のユーザーエージェントの実装。

1. 導入

1.1 スコープ

この仕様は、3つの重要な概念を定義する。

  1. 出版物を記述する一般的なマニフェストのフォーマット、
  2. マニフェストを使用してウェブ上の出版物（ウェブ出版物）を表す具体的なフォーマット、そして、
  3. ウェブへの経路を要求するモジュール式の拡張機能の基盤としてマニフェストを使用するためのルール。

この仕様は、作成できる出版物の性質を制限しようとするものではなく、ウェブ技術を使用して表現できるあらゆる種類の作品がスコープである。例えば、オーディオブックの制作など、特定の出版分野のニーズに適応できるようにも設計されており、専門的なものを作成するためにモジュール式アプローチを奨励している。

可能な限り、この仕様は既存の Open Web Platform 技術を活用し、ウェブ自体の動作方法を変更することなく、ウェブ上及びウェブの外で有効性の尺度を実現するという目標を達成する。

さらに、この仕様は、Open Web Platform 技術の更新に自動的に適応するように設計されており、（例えば、特定の日付のついたバージョンの代わりに、最新の公開バージョンを参照するなど）ウェブの進化に応じて、適合する出版物がシームレスに相互運用し続けることを保証する。

この仕様はまた、ウェブ出版物又はそこから派生した任意のフォーマットを使用するための様々なユーザー エージェント アーキテクチャを促進することを目的とする。主な目標は、従来のウェブ ユーザー エージェント（ブラウザ）が、ウェブ出版物を利用できるようにすることだが、これにより、（例えば、アプリケーション、スタンドアロンまたはユーザー エージェント内での実行や独自のユーザー インターフェイスを包含するウェブ出版物など）他の種類のユーザー エージェントの機能が制限されることはない。その結果、仕様は、ユーザーエージェントの性質と（例えば、ウェブ出版物の開始を促す方法、又はオフラインで使用するためにキャッシュするウェブ出版物のポイントや量など）ユーザーの期待に応じて予想される結果が異なる状況に必要なソリューションを設計しようとはならない。

この仕様は、ユーザー エージェントがウェブ出版物を表示する方法を定義しない。ユーザー エージェントが、ユーザーの読書体験を向上させるために提供できるアフォーダンスの種類の詳細は、[PWP-UCR] で定義される。

1.2 構成

このセクションは規定ではない。

この仕様は、三つの異なる部分から構成されており、それぞれが前の部分に基づいている。

第 I 部 — 出版物のマニフェスト

この仕様の最初の部分は、デジタル出版物に関する情報を表現するための一般的なマニフェストのフォーマットを定義する。出版物に関する様々な構造化プロパティを包含するために拡張された [schema.org] メタデータを使用し、[JSON-LD] でシリアライズされる。この定義は、デジタル出版物における全ての特定の実装の基礎となるように設計されている。表現する必要のある情報の差異に対応しながら、フォーマット間の相互運用性を実現する。デジタル出版フォーマットの実際のマニフェスト要件は、それぞれの仕様によって定義される。

第 II 部 — ウェブ出版物

この仕様の二つ目の部分は、ウェブ上の出版物を定義及び配布するためのウェブ出版物フォーマットについて説明する。例えばプライマリ エントリー ページ及び目次などの様々な実装の詳細だけでなく、ウェブ出版物のマニフェストを表現するための要件も説明する。ウェブ出版物は、出版物マニフェストに基づく全てのデジタル出版物のウェブ配布フォーマットであるため、このような拡張機能は全て、このセクションで定義されている要件との互換性を維持する必要がある。

第 III 部 — モジュール拡張

この仕様の三つ目の部分は、出版物マニフェスト モデルを使用して、新しいパッケージ化されたデジタル出版物フォーマットを作成する方法を定義する。このようなフォーマットは、ネイティブ状態のウェブ出版物に準拠する必要はないが、両方の要件に準拠できるコンテンツを作成する必要がある。

1.3 用語

このドキュメントは、特に、ユーザー、ユーザー エージェント、ブラウザー、アドレスを包含する、W3C Note 「Publishing and Linking on the Web」 [publishing-linking] で定義されている用語を使用する。

Digital Publication

デジタル出版という用語は、ウェブ出版かモジュール拡張に関わらず、この仕様に準拠する任意のフォーマットの出版物のエンコーディングを指すために使用される。このようなデジタル出版物は全て、共通のマニフェスト フォーマットを共有しているが、構造とコンテンツの要件が異なる。

Manifest

マニフェストは、例えば有益なメタデータ、全てのリソースのリスト、デフォルトの読み上げ順序など、出版物に関する構造化された情報を表現する。

Non-empty

この仕様の目的上、non-empty は、テキスト コンテンツや値が空白文字正規化後の複数の文字から成り立つ要素、属性、又はプロパティを参照するために使用され、空白文字正規化のルールは、ホスト フォーマットごとに定義される。

Web Publication

ウェブ出版物は、複数のリソースのコレクションであり、デフォルトの読み上げ順序を持つ単一の論理的な作品内のマニフェストを介して整理される。ウェブ出版物は、Open Web Platform 技術を使用して一意に識別および表示可能である。

1.4 適合性

規定ではないとマークされたセクションと同様に、この仕様書の全てのオーサリング ガイドライン、図、例及び注意は規定ではない。この仕様書の他の全ては規定である。

このドキュメントのキー ワード MAY、MUST、MUST NOT、OPTIONAL、RECOMMENDED、REQUIRED、SHOULD、SHOULD NOT は、ここに示すように BCP 14 [RFC2119] [RFC8174] で説明されているように解釈され、すべての大文字で表示される。

2. 第 I 部：出版物のマニフェスト

2.1 導入

このセクションは規定ではない。

デジタル出版物は、JSON-LD [json-ld] フォーマット（リンクトデータの JSON [ecma-404] のバリアント）を使用して表現されたプロパティのセットを提供するところのマニフェストで記述される。

マニフェストには、出版物の性質と構造に関する情報と同様に、例えばタイトルや著者など、出版物に関する表現的プロパティを包含する。

このセクションは、マニフェストの構成要件、及びマニフェストで使用するプロパティの一般的なセットの概要を説明する。

2.2 作成及び正規のマニフェスト

このセクションは規定ではない。

デジタル出版物の状態に応じて、マニフェストは次の二つ形式のいずれかで存在する。

作成されたマニフェスト

作成された出版物のマニフェストは、（すなわち、ユーザー エージェントによって処理されるデジタル出版物の前に）著者がデジタル出版物に提供するマニフェストのシリアル化である。著者は人間である必要がないことに注意されたい。機械は、自動的にデジタル出版物のために作成されたマニフェストを生成できる。

正規のマニフェスト

正規の出版物のマニフェストは、作成されたマニフェストを処理し、可能な限り全ての曖昧さを取り除き、別のソースから推測できる欠損値を組み込むとき、ユーザーエージェントによって作成されるマニフェストのバージョンである。

曖昧さや情報の欠落がない場合、作成されたマニフェストは正規のマニフェストと同等の可能性があるが、正規のマニフェストは、ユーザー エージェントが取得した処理の一部として作成されたマニフェストを検査した後にのみ存在する。

仕様のこの部分は、デジタル出版物のフォーマットに関係なく、作成されたマニフェストを制作するための要件について説明する。作成されたマニフェストから正規のマニフェストを構築するためのルールは、（例えば、ウェブ出版物のプロセスは、§ 2.9.3 正規のマニフェストの生成で説明されるなど）特定の実装ごとに定義される。

2.3 Web IDL

デジタル出版物のマニフェストは [json-ld] として作成されるが、ユーザー エージェントはプロパティを利用するために、任意の言語にすることができるところの、内部データ構造にあるこの情報を処理する。この処理が発生する正確な方法、及びデータが内部で使用される方法は、ユーザー エージェントに依存しており、この仕様では定義されない。

開発者向けのマニフェスト フォーマットの理解を単純にするため、この仕様は、Web Interface Definition Language (Web IDL) [webidl-1] — PublicationManifest 辞書を使用して、マニフェストで採用されたデータ構造の抽象表現を定義する。

この定義は、マニフェストの各要素に期待される名前、データ型、及び可能な制限を表している。しかしながら、一般的な Web IDL 定義とは異なり、ユーザー エージェントは、API としてマニフェストの情報を公開することは想定されていない。Web IDL 言語は、データモデルの抽象化を提供するためだけに選択される。

Note

デジタル出版物を作成するために、Web IDL の定義を理解する必要はない。オーサリング要件は、次のセクションで定義される。

2.4 マニフェストのコンテキスト

デジタル出版物のマニフェストは、JSON-LD コンテキスト [json-ld] を設定することから始めなければならない（MUST）。コンテキストは、次の二つの主要なコンポーネントがある。

  • [schema.org] のコンテキスト：https://schema.org
  • 出版物のコンテキスト：https://www.w3.org/ns/wp-context
例 1：コンテキストの宣言。 
{
"@context" : ["https://schema.org", "https://www.w3.org/ns/wp-context"],
    …
}

出版物コンテキストのドキュメントは、（例えば、creator プロパティが順序を維持するための要件など）Schema.org で定義されたプロパティに機能を追加する。

Editor's note

Schema.org との継続的な関係の一環として、出版物コンテキストのファイルで定義されている追加機能は、コアの Schema.org 語彙に移行できる。

Note

Schema.org はしばしば、http URI スキームを使用して参照されるが、デフォルトとして安全な https スキームを使用するように語彙は移行されてる。この仕様は、マニフェストで Schema.org を参照するとき、https を使用する必要がある。

2.5 出版物タイプ

デジタル出版物のマニフェストは、type 用語 [json-ld] を使用して出版物タイプを宣言しなければならない（MUST）。type は、CreativeWork [schema.org] にマッピングされてもよい（MAY）。

例 2：CreativeWork に出版物タイプを設定する。 
{
"@context" : ["https://schema.org", "https://www.w3.org/ns/wp-context"],
"type"     : "CreativeWork",
    …
}

Schema.org は、例えば、Article、Book、TechArticle、Course など、CreativeWork のより具体的なサブタイプも多数包含している。これらは、CreativeWork の代わりに、又は追加して使用してもよい（MAY）。

例 3：Book に出版物タイプを設定する。 
{
"@context" : ["https://schema.org", "https://www.w3.org/ns/wp-context"],
"type"     : "Book",
    …
}

各 Schema.org タイプは、その使用に適合するプロパティのセットを定義する。マニフェストが、Schema.org に対応するプロセッサによって検証および処理できることを保証するために、マニフェストは、選択されたタイプに関連付けられたプロパティのみを含めるべきである（SHOULD）。

プロパティに複数のタイプが必要な場合、マニフェストに複数のタイプ宣言を包含してもよい（MAY）。

例 4：Book と VisualArtwork の両方のプロパティを組み合わせた出版物。 
{
"@context" : ["https://schema.org", "https://www.w3.org/ns/wp-context"],
"type"     : ["Book", "VisualArtwork"],
    …
}

ユーザー エージェントは、宣言された Schema.org タイプに対して適合していないマニフェストの処理を失敗するべきではない（SHOULD NOT）。

Note

完全な CreativeWork サブタイプのリストは、Schema.org サイトを参照されたい。

2.6 プロパティ

2.6.1 導入

このセクションは規定ではない。

デジタル出版物のマニフェストは、ユーザー エージェントが出版物の処理とレンダリングに必要な基本情報を記述するプロパティのセットによって定義される。これらのプロパティは、次のように分類される。

表現的プロパティ

表現的プロパティは、例えば、タイトル、著者、言語など、デジタル出版物の外見を説明する。これらのプロパティは、主に Schema.org とホストされている拡張機能 [schema.org] から取得されるため、一つ又は複数の Schema.org プロパティにマップされ、構文とセマンティクスを継承する。（通常、次のプロパティのカテゴリは、Schema.org に相当するものがないため、出版物専用に定義される。）

リソース分類

リソース分類プロパティは、例えば、リソースのリストやデフォルトの読み上げ順序など、リソースの共通セットを記述又は識別する。これらのプロパティは、例えば、HTML ドキュメント、画像、スクリプト ファイル、個別のメタデータ ファイルなど、複数のリソースを参照する。

Note

プロパティの分類は、その目的の理解を簡単にするためだけに存在し、グループ化は、この仕様外で関係性はない（つまり、プロパティは実際にはマニフェストでグループ化されない）。

Note

schema.org から取得された各マニフェスト アイテムは、マップ先のプロパティを識別し、括弧内にその定義タイプを包含する。しかしながら、schema.org の継承モデルの結果として、プロパティはしばしば多くのタイプで利用できる。使用が妥当である場所に関する詳細は、各プロパティの定義を参照されたい。

Schema.org は、さらに、出版物に関連するものの、この仕様では言及されていない多数のプロパティを包含し、出版物の著者は、これらのプロパティのどれも使用できる。このドキュメントは、マニフェスト アイテムの最小セットのみを定義する。

Editor's note

沢山の schema.org 用語を参照し、ベスト プラクティスなドキュメントを作成するかどうかについての議論がある。その場合は、ここからリンクされているはずである。

2.6.2 値のカテゴリ

このセクションは、出版物マニフェストのプロパティに使用できる値のカテゴリを説明する。

2.6.2.1 リテラル

一部のマニフェスト プロパティは、（例えば、コード値や日付など、言語に依存しないものを）値としてリテラル テキストの文字列が必要である。これらの値は、[json] 文字列として表される。

例えば、オブジェクトに変換されるかもしれない他の値とは異なり、リテラル値はマニフェストの正規化中に変更されない。

2.6.2.2 数値

一部のマニフェスト プロパティは、値として数値が必要である。 これらの値は、[json] 数値として表される。

2.6.2.3 明示的及び暗黙的なオブジェクト

様々なマニフェスト プロパティは、[json] オブジェクトとして表現されることが期待される。通常、オブジェクトの使用が推奨されるが、コンテキストに応じてオブジェクトとして解釈される文字列値を使用することも許容できる。オブジェクトへのテキスト値の正確なマッピングは、プロパティまたはオブジェクト定義の一部である。

2.6.2.3.1 ローカライズ可能な文字列

一部のマニフェスト プロパティは、値としてローカライズ可能なテキスト文字列を期待している。 これらの値は、次のいずれかで表される。

  • 単一の文字列値、
  • プロパティのテキストとテキストの言語を識別する language プロパティを含んでいる value プロパティを持つ匿名オブジェクト。

単一の文字列値の場合、これらは、value プロパティが文字列のテキストであり、言語がマニフェスト内の他の情報から決定される暗黙のオブジェクトを表している。

2.6.2.3.2 エンティティ

出版物マニフェストのプロパティ セットの暗黙的なオブジェクトの一般的なケースは、creators 向けである。作成の様々な側面に対して責任のあるエンティティは、[schema.org] Person 及び/又は Organization オブジェクトとして表現される。しかしながら、オーサリングを簡素にするため、単純な文字列値をエンティティ名に使用できる。この場合、エンティティは Person を表すと想定される。

例 5：Person オブジェクトの代わりにテキスト文字列を使用する。

次の著者名はテキスト文字列として表現される。

{
"author" : "Herman Melville",
    …
}

しかし、creators コンテキストは、次と同等である。

{
"author" : {
"type" : "Person",
"name" : "Herman Melville"
    },
    …
}

（詳細は、§ 2.6.3.4 Creators を見よ。）

2.6.2.3.3 リンク

表現的プロパティを除き、マニフェスト プロパティは通常、複数のリソースにリンクする。プロパティにリンク値が必要な場合、リンクは次の二つの方法のいずれかで表現しなければならない（MUST）。

  1. リソースの URL をエンコードする文字列として。または、
  2. ターゲットリソースの URL、メディアタイプ、及びその他の特性を表現するために使用できる LinkedResource オブジェクトのインスタンスとして。

単一の文字列値の場合、これらは、url プロパティがその文字列値に設定されている暗黙の LinkedResource オブジェクトを表す。

例 6：相対 URL を使用し、文字列（'datatypes.svg'）としてひとつのリンクと、LinkedResource オブジェクトの様々なプロパティを表示する二つのリンクを包含するリソースのリスト 
{
    …
"resources" : [
"datatypes.svg",
        {
"type"            : "LinkedResource",
"url"             : "test-utf8.csv",
"encodingFormat"  : "text/csv",
"name"            : "Test Results",
"description"     : "CSV file containing the full data set used."
        },
        {
"type"            : "LinkedResource",
"url"             : "terminology.html",
"encodingFormat"  : "text/html",
"rel"             : "glossary"
        }
    ]
}
2.6.2.4 URL

URL は、デジタル出版物に関連付けられたリソースを識別するために使用される。これらは、妥当な URL 文字列 [url] でなければならない（MUST）。

マニフェスト URL は、http 及び https schemes [url] のみに制限される。ユーザー エージェントは、マニフェスト内の全ての URL を間接参照する必要はないが、URL はリソースを間接参照しなければならない（MUST）。

相対 URL 文字列の場合、これらは、ベース URL [url] を使用する絶対 URL 文字列に解決される。

相対 URL 文字列のベース URL は、次のように決定される。

  • 埋め込まれたマニフェストの場合、ベース URL は、出版物のプライマリ エントリ ページのドキュメント ベース URL [html] である。
  • リンクされたマニフェストの場合、ベース URL は、マニフェスト リソースの URL である。

結果として、埋め込まれたマニフェストの相対 URL 文字列は、ページがベース URL（つまり、ヘッダーの <base> 要素）を宣言しない限り、プライマリ エントリ ページの URL に対して解決される。

Note

URL は、[rfc3987] に従う Unicode 文字を使用できる。詳細は、HTML5 仕様の注 を見よ。

2.6.2.5 識別子

識別子は、永続的及び明確な方法でウェブ コンテンツを参照するために使用できる URL レコード [url] である。URL、URN、DOI、ISBN、及び PURL は、全て、出版物で頻繁に使用される永続的な識別子の例である。

2.6.2.6 配列

一部のマニフェスト プロパティは、それぞれのタイプ（リテラル、オブジェクト、又は URL）の複数の値を使用できる。一般的なルールとして、これらの値は、[json] 配列として表現できる。しかしながら、プロパティ値が、単一要素の配列である場合、配列構文は省略してもよい（MAY）。

例 7：配列の代わりにテキスト文字列を使用する

通常、デジタル出版物は、多くのリソースが含まれているため、単一のリソースの宣言は次のとおりであり、

{
"resources" : "datatypes.svg",
    …
}

配列と同等である。

{
"resources" : ["datatypes.svg"],
    …
}

2.6.3 表現的プロパティ

2.6.3.1 アクセシビリティ

アクセシビリティ プロパティは、様々な好みの読書様式を持つユーザーによって消費されるデジタル出版物の適合性に関する情報を提供する。通常、これらのプロパティは、例えば、[WCAG20] が提供しているような、確立されたアクセシビリティ基準に対する評価を補完する。（詳細なアクセシビリティ レポートへのリンクは、§ 2.7.2.1 Accessibility Report を見よ。）

次のプロパティは、アクセシビリティ プロパティとして分類される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
accessMode 個人が情報を処理又は知覚するための人間の感覚知覚システムまたは認知機能。 複数のテキスト リテラルの配列 accessMode (CreativeWork)
accessModeSufficient リソースにある全ての知的コンテンツを理解するのに十分な単一または結合された accessMode のリスト。 複数のItemList リテラルの配列 accessModeSufficient (CreativeWork)
accessibilityFeature 例えば、アクセシブルなメディア、代替手段、サポートされているアクセシビリティの強化など、リソースのコンテンツ機能。 複数のテキスト リテラルの配列 accessibilityFeature (CreativeWork)
accessibilityHazard 一部のユーザーにとって生理学的に危険であると称されるリソースの特性。 複数のテキスト リテラルの配列 accessibilityHazard (CreativeWork)
accessibilitySummary 他のアクセシビリティ メタデータと一貫しているが、例えば「短い説明はあるが、非視覚ユーザーには長い説明が必要」又は「短い説明があり、長い説明は必要ない」など、機微を表現する、特定のアクセシビリティの機能や欠陥の人間が可読可能な要約。 テキスト ローカライズ可能な文字列 accessibilitySummary (CreativeWork)
Note

それらのプロパティで使用する予定の値を包含する、これらのプロパティの詳細な説明は、[webschemas-a11y] が利用できる。

Note

著者は、これらのプロパティで表現できる以上の情報が必要な場合、詳細なアクセシビリティ レポートへの参照も提供できる。

例 8：テキストと画像を含むドキュメントのアクセシビリティ メタデータの例。出版物は、各画像に適した代替テキストと長い説明を提供するため、純粋なテキスト形式で読むこともできる。 
{
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"     : "CreativeWork",
    …
"accessMode"            : ["textual", "visual"],
"accessModeSufficient"  : [
        {
"type"           : "ItemList",
"itemListElement": ["textual", "visual"]
        },
        {
"type"           : "ItemList",
"itemListElement": ["textual"]
        }
    ],
    …
}
2.6.3.2 アドレス

デジタル出版物のアドレスは、プライマリ エントリ ページを識別する URL である。それは、url プロパティを使用して表現される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
url プライマリ エントリ ページの URL 妥当な URL 文字列 [url]。 URL の配列 url (Thing)

デジタル出版物は、ひとつ以上のアドレスを持ってもよいが（MAY）、全てのアドレスは、同じドキュメントに解決されなければならない（MUST）。

Note
出版物のアドレスは、識別子のリンク関係 [link-relation] の値としても使用できる。
例 9 ：メイン エントリ ページのアドレスの設定 
{
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"     : "Book",
    …
"url"      : "https://publisher.example.org/mobydick",
    …
}
2.6.3.3 正規の識別子

デジタル出版物の正規識別子プロパティは、出版物の一意の識別子を提供する。それは、id プロパティを使用して表現される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
id 出版物の優先バージョン。 URL レコード [url] Identifier (None)

正規識別子は、デジタル出版物の優先バージョンを解決する URL となるべきである（SHOULD）。URL を使用すると、デジタル出版物のアドレスを超えた永続性の基準を提供する。例えば、デジタル出版物が永続的に新しい URL に再配置された場合、（例えば、DOI レジストリは、新しい URL で更新したり、リダイレクトは、正規識別子の URL 追加できるなど）正規のアドレスは新しい位置を発見する方法を提供する。異なる URL でホストされている同じデジタル出版物のインスタンスを識別する手段を提供することも意図している。

Note

正規識別子の一意性を保証することは、この仕様のスコープ外である。実際に達成可能な一意性は、使用される識別子スキームの規則や識別子の割り当ての制御の程度などの要因に依存する。

正規識別子が、マニフェストに提供されていない、又は値が無効な URL である場合、デジタル出版物は正規識別子を持たない。ユーザー エージェントは、正規のマニフェストに対するマニフェストに提供される他の識別子から正規の識別子を構築しようとしてはならない（MUST NOT）。

正規の識別子の仕様は、identifier プロパティ [schema.org] 及び/又はそのサブタイプを使用して、追加の識別子のタイプを包含することで補完してもよい（MAY）。

例 10：URL として正規の識別子とアドレスを設定する例 
{
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"     : "TechArticle",
    …
"id"       : "http://www.w3.org/TR/tabular-data-model/",
"url"      : "http://www.w3.org/TR/2015/REC-tabular-data-model-20151217/",
    …
}
例 11：正規識別子の URN の例 
{
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"     : "Book",
    …
"id"       : "urn:isbn:9780123456789",
"url"      : "https://publisher.example.org/mobydick",
    …
}
2.6.3.4 クリエイター

creator は、デジタル出版物の作成を担当する個人又は団体である。

次のプロパティは、クリエイターとして分類される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
artist 鉛筆又はデジタル線画以外の媒体における出版物の主要なアーティスト。 複数の Person エンティティの配列 artist (VisualArtwork)
author 出版物の著者。 複数の Person 及び/又は Organization エンティティの配列 author (CreativeWork)
colorist インクの描画に色を追加する人。 複数の Person エンティティの配列 colorist (VisualArtwork)
contributor 役割がこの表の他の役割のいずれにも適合しない貢献者。 複数の Person 及び/又は Organization エンティティの配列 contributor (CreativeWork)
creator 出版物の作成者。 複数の Person 及び/又は Organization エンティティの配列 creator (CreativeWork)
editor 出版物の編集者。 複数の Person エンティティの配列 editor (CreativeWork)
illustrator 出版物のイラストレーター。 複数の Person エンティティの配列 illustrator (Book)
inker インクで鉛筆画をトレースする人。 複数の Person エンティティの配列 inker (VisualArtwork)
letterer アートワークに吹き出しや効果音などのレタリングを追加する人。 複数の Person エンティティの配列 letterer (VisualArtwork)
penciler 主要な物語のアートワークを描く人。 複数の Person エンティティの配列 penciler (VisualArtwork)
publisher 出版物の出版社。 複数の Person 及び/又は Organization エンティティの配列 publisher (CreativeWork)
readBy （オーディオブック用の）出版物を読む（演じる）人。 複数の Person エンティティの配列 readBy (Audiobook)
translator 出版物の翻訳者。 複数の Person 及び/又は Organization エンティティの配列 translator (CreativeWork)

クリエイターは、次の二つの方法のいずれかで表される。

  1. Person [schema.org] の名前をエンコードした文字列として、または、
  2. Person 又は Organization [schema.org] のインスタンスとして。

言い換えると、単一の文字列値は、name プロパティがその文字列値に設定されている [schema.org] Person の省略形である。（§ 2.6.2.3.2 Entities もまた見よ。）

[schema.org] Person 又は Organization タイプから creator information の各セットをコンパイルするとき、ユーザー エージェントは、利用可能なとき、次の情報を保持しなければならない（MUST）。

type
クリエイターのタイプを識別する複数の文字列。このシーケンスは、「Person」または「Organization」を包含するべきである（SHOULD）。
name
クリエイターの名前のための複数のローカライズ可能な文字列。
id
識別子としてのクリエイターの正規の識別子。
url
URL 形式のクリエイターのアドレス。

ユーザー エージェントは、Schema.orgで定義されたクリエーター プロパティの範囲を、上記のリストよりも広く解釈してもよい（MAY）ことに注意されたい。

マニフェストは、各タイプのクリエイターが複数含まれてもよい（MAY）。

例 12：本の著者 
{
"type"     : "Book",
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
    …
"url"      : "https://publisher.example.org/mobydick",
"author"   : {
"type"  : "Person",
"name"  : "Herman Melville"
    }
}
例 13 ：単純な文字列として一部の人が表現されている、編集者、著者、出版社の個別のリスト。 
{
"@context"   : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"       : "TechArticle",

2.6.3.6 言語と基本方向

デジタル出版物は、（例えば、英語、フランス語、中国語など）コンテンツを表現する言語である、自然言語を少なくともひとつ持つ。また、書字するための自然な基本方向 (左から右かそれとも右から左への表示方向) もある。

デジタル出版物のマニフェストは、（例えば、自然な基本方向が右から左である出版物の場合、右側に目次のポップアップを配置されるなど）ユーザー エージェントの動作などに影響を与えるこれらの両方の概念を設定するためのエントリを包含する。

Note

それを構成する個々のリソースの言語及び基本方向から、出版物の言語を区別することは重要である。例えば、そのようなリソースが HTML にある場合、言語と方向もこれらのリソースに設定する必要がある。出版物の言語と基本方向は継承されない。

同様に、（例えば、タイトル、クリエイターなど）マニフェスト内の各自然言語プロパティ値は、ローカライズ可能な文字列である。

Note

ウェブ上のローカライズされた文字列の詳細は、[string-meta] を参照されたい。

自然言語と基本方向は、出版物とマニフェストの自然言語プロパティ値の両方に設定できる。

ユーザー エージェントが、言語を必要とし、（グローバルかそれともそのプロパティ専用の）作成されたマニフェストで使用できない、又は取得された値が無効な場合、ユーザー エージェントは、正規のマニフェストを生成するときに言語を決定してもよい（MAY）。この仕様は、そのような言語タグの作成方法を強制しない。ユーザー エージェントは次のことを行う。

  • マニフェストの空ではない言語の宣言を使用し、
  • デフォルトの読み上げ順序で、リソースにある最初の空でない言語宣言を使用し、
  • 独自のアルゴリズムを使用して言語を計算する。

言語又はデフォルトの基本方向のデフォルト値は指定されない。

2.6.3.6.1 全体の言語と方向

マニフェストには、次のプロパティを使用して、出版物の全体の言語及び基本方向の宣言を包含してもよい（MAY）。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
inLanguage テキストのマニフェストだけでなく出版物のデフォルト言語の値。 [bcp47] で定義されている言語コード リテラル inLanguage (Property)
inDirection テキストのマニフェストだけでなく出版物のデフォルトの基本方向の値。 ltr、rtl、又は auto リテラル (None)

自然言語は、[bcp47] に準拠するタグでなければならず（MUST）。基本言語の方向は、次の値のいずれかを持たなければならない（MUST）。

  • ltr：テキスト値が、明示的に方向に関して左から右のテキストに設定されていることを示す。
  • rtl：テキスト値が、明示的に方向に関して右から左のテキストに設定されていることを示す。
  • auto：Unicode 双方向アルゴリズム [bidi] の規則に従って、テキスト値が強い指向性を持つ最初の文字の方向に、明示的な指向性を設定されることを示す。

指定したとき、これらのプロパティは、マニフェストのテキスト値のデフォルトとしても使用される。

全体の言語情報は、個々の値によって上書きされてもよい（MAY）。

マニフェストが、script 要素を介してプライマリ エントリ ページに埋め込まれ、マニフェストが、全体の言語及び/又は基本方向を設定しない場合（§ 2.6.3.6.1 Global Language and Direction を見よ）、それぞれ、script 要素の lang と dir 属性が全体の言語及び基本方向として使用される（[html] の lang 及び dir 属性の詳細を見よ）。

2.6.3.6.2 項目固有の言語

マニフェスト内の任意のテキスト値に、言語を設定できる。この情報は、localizable string として、つまり、（単純な文字列の代わりに）value 及び language の用語 [json-ld] を使用して設定しなければならない（MUST）。

例 15：localizable string を使用して著者名をフランス語に設定する 
{
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"     : "Book",
    …
"author" : {
"type"  : "Person",
"name" : {
"value"    : "Marcel Proust",
"language" : "fr"
        }
    }
}

language の値は、[bcp47] で定義されている言語コードに設定しなければならない（MUST）。

ローカライズ可能なテキストのコンテキストで使用される場合、単純な文字列値は localizable string のショートハンドであり、value は文字列値に設定され、言語は該当する場合は inLanguage プロパティの値に設定され、それ以外の場合は設定されない。言い換えると、上記の例は次と同等である。

例 16：フランス語にデフォルトの著者名のデフォルト言語を設定する 
{
"@context"   : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"       : "Book",
"inLanguage" : "fr",
    …
"author"     : "Marcel Proust",
    …	
}

（§ 2.6.2.3 Explicit and Implied Objects も見よ。）

値に対して方向を明示的に設定することはできない。

Note

自然なテキスト値の方向を設定することは、現在、JSON-LD [json-ld] では不可能である。schema.org コミュニティと同様に JSON-LD コミュニティがこのような機能を導入した場合、この仕様の将来のバージョンでは、マニフェストの機能を拡張してこれを包含するかもしれない。

右から左又は双方向のテキストを含むマニフェスト エントリを正しく処理するために、ユーザー エージェントは、最初の強い指向性のある文字のテキストをスキャンすることで、特定の自然言語値の基本方向を識別するべきである（SHOULD）。

Note

最初の強力なヒューリスティックが、（例えば、ラテン語の頭字語で始まるアラビア語又はヘブライ語のスクリプトの文字列など）誤った結果を生成する状況では、コンテンツ開発者は、文字列に対して Unicode フォーマットの文字を追加することができる。これにより、ヒューリスティックが適用されるとき、必要な基本方向が生成される。次のフォーマット文字のいずれかを使用する必要がある：U+200E LEFT-TO-RIGHT MARK、又は U+200F RIGHT-TO-LEFT MARK。（§ D. Examples for bidirectional texts を見よ。）

基本方向が識別されると、ユーザー エージェントは、Unicode 双方向アルゴリズム [bidi] に従って、自然言語値の適切なレンダリングと表示を決定しなければならない（MUST）。これは、基本方向を適用するため、表示する前に文字列の周りに追加のマークアップ又は Unicode フォーマットの文字をラップする必要がある。

2.6.3.7 最終変更日

最終変更日は、デジタル出版物が最後に更新されたときの日付である（つまり、マニフェストを包含する出版物のリソースに最後に変更が加えられたとき）。dateModified プロパティを使用して表現される。

2.6.3.9 組み方向

reading progression は、デジタル出版物内のあるリソースから次のリソースへの読書方向を確立する。readingDirection プロパティを使用して表現される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
readingProgression あるリソースから他のリソースへの読書方向。 ltr 又は rtl リテラル (None)

このプロパティの値は、次のいずれかでなければならない（MUST）。

  • ltr：左から右
  • rtl：右から左

デフォルト値は、ltr である。

このプロパティは、個々のプライマリ リソースのレンダリングに影響しない。 あるリソースから別のリソースへの進行方向にのみ関係する。

Note

出版物の組み方向は、メニュー位置、スワップ方向、ユーザーを次のページや前のページに導くためのタップゾーンの定義、タッチジェスチャなどの出版物レベルのインタラクションを適応させるために使用される。

readingProgression が設定されていない場合、ユーザー エージェントは、正規のマニフェストを生成するとき、デフォルト値 ltr を使用しなければならない（MUST）。

例 19：組み方向が明確に ltr に設定 
{
"@context"           : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"               : "Book",
    …
"url"                : "https://publisher.example.org/mobydick",
"readingProgression" : "ltr"
}

タイトルは、デジタル出版物の人間が可読可能な名前を提供する。name プロパティを使用して表現される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
name 出版物の人間が可読可能なタイトル。 タイトルのための複数の項目。 ローカライズ可能な文字列の配列 name (Thing)

2.6.4 リソース分類プロパティ

出版物のリソースは、このセクションで定義されているデフォルトの読み上げ順序、リソースのリスト、及びリンクを介して指定される。これらの一覧は、プライバシー ポリシーなどの情報リソースへの参照、及び目次などの構造リソースへの参照が含まれる。

特定のリソースの URL がこれらの一覧に複数現れてはならず（MUST NOT）、URL は、リスト内で繰り返してはならない（MUST NOT）ことに注意されたい。

マニフェストは、これらのリスト内に自体の参照を包含してはならない（MUST NOT）。

2.6.4.1 デフォルトの読み上げ順序

デフォルトの読み上げ順序は、デジタル出版物リソースのセットを経由する特定の進行順序である。ユーザーは、コンテンツを通る経由する代わりの経路を辿る場合があるが、そのようなインスタンスが欠如している場合、デフォルトの読み上げ順序は、あるリソースから次のリソースへの予想される進行順序を定義する。

デフォルトの読み上げ順序は、readingOrder プロパティを使用して表現する。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング

  • LinkedResource オブジェクトの複数のインスタンス

項目の順序は重要でる。URLs は、フラグメント識別子を包含してはならない（MUST NOT）。非 HTML リソースは、encodingFormat 値が設定された LinkedResource オブジェクトとして表現されるべきである（SHOULD）。

 リンクの配列 (None)

2.6.4.2 リソース リスト

リソース リストは、デフォルトの読み上げ順序に掲載されていないデジタル出版物の処理とレンダリングに使用される追加リソースを列挙する。resources プロパティを使用して表現される。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
resources
  • リソースの URL を表現する複数の文字列、又は、
  • LinkedResource オブジェクトの複数のインスタンス

 リンクの配列 (None)

リソース リストの完全性は、（例えば、オフラインで読む機能など）特定の読書状況におけるデジタル出版物のユーザビリティに影響する可能性がある。このため、デフォルトの読み上げ順序に掲載されているリソース以外の、パブリケーション内にある全ての構成リソースの包括的なリストを提供することを強く推奨する（RECOMMENDED）。

2.6.4.3 リンク

links プロパティは、デジタル出版物の処理とレンダリングに必要ないリソースのリストを提供する（つまり、これらのリソースが利用できない場合でも、出版物の内容は影響を受けない）。

用語 概要 必須の値 値のカテゴリ [schema.org] マッピング
links
  • URL を表現する複数の文字列、又は、
  • LinkedResource オブジェクトの複数のインスタン

項目の順序は、重要ではない。encodingFormat 及び rel 値を設定した LinkedResource オブジェクトの使用を推奨する（RECOMMENDED）。

 リンクの配列 (None)

通常、リンクされたリソースは、処理又はレンダリングを補強又は強化するためにユーザー エージェントを利用可能にする。例えば、

  • ユーザー エージェントが、シェルフからリンクを提示できるプライバシー ポリシー又はライセンス、
  • ユーザー エージェントが、出版物に関する詳細情報を発見及び表示するために使用できるメタデータ レコード、
  • ユーザー エージェントが、拡張された言語のヘルプを提供するために処理ができる用語の辞書。

リンクは、出版物のオンライン レンダリングで使用されるリソースを識別することもできるが、（例えば、サイズを最小化するなど）出版物がオフラインやパッケージ化されたときに包含する必要はない。これらは以下が含まれる。

  • 出版物の見た目を強化するが、表示には不可欠ではない巨大なフォント ファイル（つまり、代替フォントで十分だろう）、
  • 出版物がオフライン又はパッケージ化されたときに使用することを意図していない（例えば、トラッキングスクリプトなど）サードパーティのスクリプト。

links のリストは、（例えば、スクリプト、画像、スタイル シートなど）リンクされたリソースをレンダリングに必要なリソースを包含するべきである（SHOULD）。

links に掲載されたリソースは、デフォルトの読み上げ順序又はリソース リストに掲載してはならない（MUST NOT）。

ユーザー エージェントは、リンクされたリソースを無視してもよく（MAY）、出版物でそれらをオフラインにする必要はない。これらのリソースは、出版物をパッケージングするときに包含するべきではない（SHOULD NOT）。

2.6.5 拡張性

マニフェストは、デジタル出版物の表示とレンダリングにおいてユーザー エージェントによって使用されるための基本的なプロパティのセットを提供するように設計されていますが、次の方法で拡張してもよい（MAY）。

  1. リンクされたメタデータ レコードの提供によって
  2. マニフェストに追加プロパティの包含を介して

どちらの方法も妥当だが、リンクされたレコードの使用を推奨する（RECOMMENDED）。

この仕様は、追加プロパティがマニフェストの内部表現においてユーザー エージェントによるコンパイル、保存、公開する方法を定義しない。ユーザー エージェントは、一部又は全ての拡張プロパティを無視してもよい（MAY）。

2.6.5.1 リンクされたレコード

例えば、ONIX [onix] 又は BibTeX [bibtex] ファイルなど、レコードへのリンクを介してマニフェストを拡張するには、LinkedResource オブジェクトを使用して表現しなければならない（MUST）。

  • LinkedResource の rel 値は、IANA 又は他の組織によって定義された関連する識別子を包含するべきであり（SHOULD）、リンク レコードに記述的なメタデータが含まれる場合、describedby（IANA）識別子を包含しなければならない（MUST）。
  • リンク内の encodingFormat 値は、該当する場合、その特定のタイプのレコードに対して定義された MIME メディア タイプ [rfc2046] を使用しなければならない（MUST）。

リンクされたレコードが、出版物の一部である場合、リソース リストに包含しなければならない（MUST）（つまり、明示的なマニフェスト以上のものに必要である）。それ以外の場合、リンク リストに包含しなければならない（MUST）。

例 24 ：本のメタデータ ファイルのための外部 ONIX へのリンク 
{
"@context"   : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"       : "Book",
…
"url"        : "https://publisher.example.org/mobydick",
"name"       : "Moby Dick",
"links"  : [{
"type"            : "LinkedResource",
"url"             : "https://www.publisher.example.org/mobydick-onix.xml",
"encodingFormat"  : "application/onix+xml",
"rel"             : "describedby"
},{
	…
}],
…
}
Editor's note

application/onix+xml MIME タイプは、このドキュメントの執筆時点ではまだ IANA によって登録されておらず、説明だけを目的として例に包含されている。

2.6.5.2 マニフェストの追加プロパティ

追加のプロパティは、マニフェストに直接包含できる。これらのプロパティは、[schema.org] や [dcterms] のようなパブリック スキームから取得し、可能な限りコントロールされた語彙から値を使用することを推奨する（RECOMMENDED）。プロプライエタリの用語を使用してもよいが（MAY）、コンテキストの一部として定義されたプレフィックスと伴に、そのような用語を、Compact IRIs [json-ld] を使用して包含することを推奨する（RECOMMENDED）。

2.7 リソースの関係

2.7.1 導入

このセクションは規定ではない。

マニフェストは、リンク リレーションを使用してデジタル出版物の主要なリソースを識別する。これらの関係は、（例えば、目次とリソース リストで見つかるリンクなど）LinkedResource オブジェクトの rel プロパティに適用される。

これらの関係が識別するリソースのタイプは、次のように分類される。

有益なリソース

有益なリソースは、例えば、プライバシー ポリシー、アクセシビリティ レポート、又はプレビューなど、出版物に関する追加の情報を含むリソースである。

構造リソース

構造リソースは、例えば、カバー画像、目次、ページ リストなど、出版物の主要なメタ構造である。

2.7.2 有益なリソース

2.7.2.1 アクセシビリティ レポート

アクセシビリティ レポートは、様々な好みの読書方法を持つユーザーが消費するためのデジタル出版物の適合性に関する情報を提供する。通常、これらのレポートは、例えば、[WCAG21] で提供されているものなど、確立されたアクセシビリティ基準に対する評価の結果を識別し、出版物のユーザビリティを決定する重要な情報源である。

アクセシビリティ レポートは、accessibility-report リンク リレーションを使用して識別する。

Editor's note

現在、accessibility-report 用語は、IANA のリンク リレーションに登録されていないが、ワーキング グループはそれを追加する予定である。

マニフェストは、出版物が利用できるとき、アクセシビリティ レポートへのリンクを包含するべきである（SHOULD）。レポートを、出版物のリソースとして包含することが推奨される（RECOMMENDED）。

アクセシビリティ レポートは、例えば、[html] など、人間が可読可能なフォーマットで提供されることも推奨される（RECOMMENDED）。例えば、Schema.org [schema.org] で提供するなど、機械処理可能なメタデータでこれらのレポートを増補することも、推奨される（RECOMMENDED）。

2.8 関連性

2.8.1 リンク

マニフェストへのリンクは、次のフォーマットのひとつ又は両方を使用しなければならない（MUST）。

  • rel のパラメーターに値「publication」が設定された HTTP Link ヘッダー フィールド [rfc5988]。

    例 36 
    Link: <https://example.com/webpub/manifest>; rel=publication

  • rel 属性に値「publication」が設定された link 要素 [html]。

    例 37 
    <link href="https://example.com/webpub/manifest" rel="publication"/>

マニフェストが、HTML ドキュメントに埋め込まれているとき、リンクは、マニフェストを含む script 要素を参照するフラグメント識別子を包含しなければならない（MUST）（§ 2.8.2 Embedding を見よ）。

例 38：同じ HTML リソース内のマニフェストへのリンク 
<link href="#example_manifest" rel="publication">
…
<script id="example_manifest" type="application/ld+json">
{
"@context" : ["https://schema.org", "https://www.w3.org/ns/wp-context"],
       …
}
</script>
Issue 132 : Using rel="publication" topic:manifest

rel の正確な値は、まだ合意されていないので、IANA によって登録される必要がある。

Editor's note

次の詳細は、将来のドラフトにおいてライフサイクル セクションに移動するかもしれない。

リソースが、複数のマニフェストにリンクするとき、ユーザー エージェントは、エンド ユーザーに複数の代替手段を提示を選択するか、又は勝手に単一の代替手段を選択してもよい（MAY）。ユーザー エージェントは、明示的に親として掲載されていなくても、（例えば、帯域外で計算又は取得する情報に基づいてなど）保有する情報に基づいてマニフェストを提示することを選択してもよい（MAY）。ユーザー エージェントの実装者による優先が欠如している場合、最初に掲載されたマニフェストの選択が、デフォルトとして提案される。

2.8.2 埋め込み

マニフェストが、HTML ドキュメント内に埋め込まれているとき、type 属性に application/ld+json が設定されている script 要素 [html] に包含しなければならない（MUST）。

さらに、script 要素は、id 属性 [html] に一意識別子を包含しなければならない（MUST）。この識別子は、マニフェストが参照できることを保証する。

例 39：ウェブ出版物のマニフェストは、HTML ドキュメントに包含している。 
<script id="example_manifest" type="application/ld+json">
   {
      …
   }
</script>

2.9 出版物マニフェストのライフサイクル

2.9.1 導入

このセクションは規定ではない。

このセクションは、含まれるデータ構造の内部表現にある作成されたマニフェストを処理するために従うユーザー エージェントの手順を説明する。

このプロセスの最初のステップはマニフェストを取得することであり、通常、これを行う正確な手順は、各デジタル出版フォーマットによって定義される。

このプロセスは、（例えば、マニフェストが script タグの中に埋め込まれている場合、情報は含んでいる HTML ドキュメントから収集されるなど）明示的に作成されていない欠落したデータを追加する表現であるマニフェストの正規形式の生成を含む。

正規のマニフェストが生成された後、データは妥当性をチェックするために後処理手順の最終セットを介して評価され、最終的にユーザーエージェントが使用できるデータ構造になる。

このプロセスの中は、独自の専門的なニーズとオーディエンスに対する基本的な要件を強化するためのデジタル出版物フォーマットにすることができる様々な拡張ポイントがある。

2.9.2 マニフェストの処理

マニフェストを処理する手順は、次のアルゴリズムによって与えられる。成功した場合、アルゴリズムは、処理されたマニフェストを返し、それ以外の場合は、途中で終了して何も返さない。何も返されない場合、ユーザー エージェントはマニフェスト宣言を無視しなければならない（MUST）。

アルゴリズムは次の引数を取る。

  • text：マニフェストに含まれている UTF-8 文字列、
  • base：マニフェストのベース URL を表す URL 文字列。
  • document：マニフェストを参照するドキュメントの HTML ドキュメント（DOM）ノード。
  1. json を、text の構文解析の結果とする。構文解析がエラーを投げた場合、このアルゴリズムを終了する。
  2. タイプ（json）が、Object でない場合、このアルゴリズムを終了する。
  3. canonical manifest は、§ 2.9.3 Generating a Canonical Manifest で説明されているアルゴリズムへの入力として、json、base、及び document 値を使用し、json から得られた正規のマニフェストとする。

  4. canonical manifest が、出版物マニフェストの最小要件を満たしているかどうかを確認する、すなわち、

    • JSON-LD コンテキストは、§ 2.4 Manifest Contexts の要求のとおりに設定され、
    • 出版物のタイプは、§ 2.5 Publication Types の要求のとおりに設定され、そして、
    • 拡張要件は、それぞれの仕様で定義されているとおりに設定される。

    これらの要件のいずれかが満たされていない場合、アルゴリズムを終了する。

  5. processed manifest は、canonical manifest が与えられた正規のマニフェストの後処理の結果とする。
  6. processed manifest を返す。
Note

このアルゴリズムは、エラーおよび警告メッセージの報告方法について説明しない。これは実装に依存する。

2.9.3 正規のマニフェストの生成

出版物マニフェストを正規のマニフェストに変換する手順は、次のアルゴリズムによって与えられる。アルゴリズムは次の引数を取る。

  • manifest：マニフェストを表す JSON オブジェクト
  • base：マニフェストのベース URL を表す URL 文字列
  • document： マニフェストを参照するドキュメントの HTML ドキュメント（DOM）ノード。

アルゴリズムの手順を次で説明する。アルゴリズムは、P が manifest かそれとも（例えば、Person など）manifest 内に現れるオブジェクトで、P["term"] がラベル "term" のオブジェクト P の値を参照するという点において、厳密な JavaScript 表記とは異なる。アルゴリズムは、manifest のためにいくつかの用語を置換又は追加し、置換する用語は、{"term":"value"} として JSON 構文で表される。

  1. lang 文字列は、デフォルトの言語を表し、次のように設定される。

    • （設定されているとき）document 内の script 要素の lang 値の値、または、
    • そうでなければ undefined
    説明

    この値は、次の言語の手順で使用される。

  2. dir 文字列は、基本方向を表し、次のように設定される。

    • （設定されているとき）document 内の script 要素の dir 値の値、または、
    • そうでなければ undefined
    説明

    この値は、次の基本方向の手順で使用される。

  3. manifest["name"] が undefined の場合の（§ 2.6.3.10 タイトル）は、（設定されているとき）document を使用して title 要素 [html] を探す）。その要素が存在し、空でない場合、t はそのテキスト コンテンツであるとして、manifest に追加する。

    • title の言語が、明示的に l の値に設定されている場合、追加する。
      "name": [{"value": t, "language": l}]
    • そうでなければ
      "name": [t]
    説明

    この手順は、name プロパティがマニフェストに指定されていないとき、document の title 要素のコンテンツを追加する。例えば、

    例 40 
    <html>
<head>
<title>Moby Dick</title>
    …
<script type="application/ld+json">
    {
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
        …
    }
</script>

    成果物。

    例 41 
    {
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"name"     : ["Moby Dick"],
    …
}

  4. manifest["inLanguage"] が undefined で、lang の値が undefined でない場合の（§ 2.6.3.6 言語と基本方向）は、
    manifest に
    "inLanguage": lang
    を追加する。

    説明

    この手順は、document へ明示的に設定された言語の妥当性を保証する。例えば、

    例 42 
    <html>
<head>
    …
<script type="application/ld+json" lang=ur>
    {
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
        …
    }
</script>

    （言語と方向がマニフェストで明示的に設定される場合を除いた）成果物。

    例 43 
    {
"@context"    : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"inLanguage"  : "ur",
    …
}

  5. manifest["inDirection"] が undefined で、dir の値が、undefined でない場合の（§ 2.6.3.6 言語と基本方向）は、
    manifest に
    "inDirection": dir
    を追加する。

    説明

    この手順は、埋め込みドキュメントへ明示的に設定された基本方向を保証する。例えば、

    例 44 
    <html>
<head>
    …
<script type="application/ld+json" dir=rtl lang=ur>
    {
"@context" : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"inLanguage"  : "ur",
        …
    }
</script>

    （言語と方向がマニフェストで明示的に設定される場合を除いた）成果物。

    例 45 
    {
"@context"    : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"inLanguage"  : "ur",
"inDirection" : "rtl"
    …
}

  6. manifest["readingOrder"] が undefined で、document.URL の値を u とする場合の（§ 2.6.4.1 デフォルトの読み上げ順序）は、
    manifest に
    "readingOrder": [{"type": ["LinkedResource"], "url": u}]
    を追加する。

    説明

    デジタル出版物が参照しているドキュメントのみで構成されている場合、デフォルトの読み上げ順序は省略され、自動的に、単一のリソースで構成される。

  7. 単一の文字列又はオブジェクトである P["term"] の各値 v に対する（§ 2.6.2.6 配列）は、term に配列を想定している場合、関連する用語／値のペアを、
    "term": [v]
    に変換する

    説明

    多くの用語は、値が配列である必要があるが、便宜上、著者は、ひとつの要素配列の代わりに単一の値を使用できる。例えば、

    例 46 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"name"      : "Moby Dick",
"author"    : "Herman Melville",
"resources" : [{
"type"           : "LinkedResource",
"rel"            : "cover",
"url"            : "images/cover.jpg",
"encodingFormat" : "image/jpeg"
    },
        …
    }],
    …
}

    成果物。

    例 47 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"name"      : ["Moby Dick"],
"author"    : ["Herman Melville"],
"resources" : [{
"type"           : ["LinkedResource"],
"rel"            : ["cover"],
"url"            : "images/cover.jpg",
"encodingFormat" : "image/jpeg"
    },
        …
    }],
    …
}

  8. 単一の文字列又はローカライズ可能な文字列である manifest["term"] 配列の各値 v に対する（§ 2.6.3.4 クリエイター）は、term にエンティティを想定している場合、配列内の要素を、
    {"type": ["Person"], "name": [v]}
    に変換する。

    説明

    著者、編集者などは、タイプ Person のオブジェクトとして明示的に設計する必要があるが、便宜上、著者は名前を指定するだけである。例えば、

    例 48 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"name"      : ["Moby Dick"],
"author"    : ["Herman Melville"],
    …
}

    成果物。

    例 49 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"name"      : ["Moby Dick"],
"author"    : [{
"type" : ["Person"],
"name" : "Herman Melville"
    }],
    …
}

  9. 単一の文字列である manifest["term"] 配列の各値 v に対する（§ 2.6.2.3.3 リンク）は、term がリソース分類プロパティのひとつであり、配列の要素を、
    {"type": ["LinkedResource"], "url": v}
    に変換する。

    説明

    リソース リンクは、タイプ LinkedResource のオブジェクトとして明示的に設計する必要があるが、便宜上、著者は絶対 URL 又は相対 URL を指定するだけである。例えば、

    例 50 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
    …
"resources" : [
"css/mobydick.css",
        …
    ],
    …
}

    成果物。

    例 51 
    {
"@context"  : ["https://schema.org","https://www.w3.org/ns/wp-context"],
    …
"resources" : [{

    },
        …
    ],
    …
}

  10. P["term"] の各値 v に対する（§ 2.6.2.3.1 Localizable Strings）、又は後者が配列である場合の P["term"] において、それは単一の文字列であり、term はローカライズ可能な文字列を想定しているならば、関連する用語／値を、

    • manifest[inLanguage] は、l の値がセットされ、その次に、
      "term": {"value": v,"language": l}
    • さもなければ
      "term": {"value": v}

    に変換する。

    説明

    自然言語のテキスト値は、ローカライズ可能な文字列オブジェクトとして明示的に設計する必要があるが、便宜上、著者は単一の文字列のみを使用できる。つまり、言語情報が、マニフェストで（inLanguage を介して）提供されていない場合、例えば、

    {

"name"      : ["Moby Dick"],
"author"    : ["Herman Melville"],
    …
}

    成果物。

    例 53 
    {

"value" : "Moby Dick"
    }],

    }],
    …

    例 54 
    {

  11. 絶対 URL 文字列ではない P["term"] の各値 v に対する（§ 2.6.2.4 URLs）、及び term が URL を想定する場所で、この値を解決し、相対 URL と見なされ、base の値の使用と au の値を生成し、用語/値のペアを
    "term": au
    で置き換える。

    説明

    出版物マニフェスト内の全ての相対 URLs は、絶対 URLs を生成するためにベース値に対して解決する必要がある。

2.9.4 正規のマニフェストの後処理

正規のマニフェストを後処理する手順は、次のアルゴリズムによって与えられる。このアルゴリズムは、正規のマニフェストを表現する json オブジェクトを受け取る。このアルゴリズムに JSON オブジェクトを入力すると、処理されたマニフェストが出力される。このアルゴリズムの目的は、json で表されるデータが、データの最小要件を満たしていることを確認し、該当する場合は、適合しないデータを削除することである。

表記の乱用としては、P["term"] は、ラベル「"term"」のオブジェクト P の値を参照し、P は、manifest、又は（例えば、LinkedResource など）manifest 内に現れるオブジェクトのいずれかである。

  1. manifest は、PublicationManifest 辞書に 変換する json の結果とする。

  2. manifest にデータ クリーンアップ操作を実行し、場合によってはデータを削除し、イシューを発する。

    1. エンティティを想定する全ての term について、manifest[term] の値オブジェクト P に、P["name"] が設定されているかどうかをチェックする。そうでない場合、manifest[term] 配列から P を削除し、イシューを発する。
    2. § 2.6.4 リソース分類プロパティで定義されている全ての term について、manifest[term] の全てのオブジェクト P に、P["url"] が設定されているかどうかをチェックする。そうでない場合、manifest[term] 配列から P を削除し、イシューを発する。そうである場合、P["url"] が妥当な URL [url] であるかどうかを確認し、そうでない場合はイシューを発する。
    3. タイプ LinkedResource の全てのオブジェクト P について、P["length"] の値が設定されている場合、この値が有効な数値であるかどうかをチェックする。もしチェックが失敗した場合、イシューを発する。
    4. manifest["name"] の値が、空ではないかどうかをチェックする。そうである場合、値を生成し（詳細については個別の注を見よ）、イシューを発する。
    5. manifest["datePublished"] が、[iso8601] に従う妥当な日付又は日時かどうかをチェックする。もしチェックに失敗した場合、イシューを発する。
    6. manifest["dateModified"]が、[iso8601] に従う妥当な日付又は日時かどうかをチェックする。もしチェックに失敗した場合、イシューを発する。
    7. manifest["duration"] が、[iso8601] に従う妥当な継続時間値かどうかをチェックする。もしチェックに失敗した場合、イシューを発する。

  3. 拡張ポイント：アルゴリズムのこの時点で、プロプライエタリ、プロファイル固有、及び/又は他のサポートされているメンバーを処理する。

  4. manifest に戻る。

3. 第 II部：ウェブ出版物

3.1 導入

このセクションは規定ではない。

ウェブ出版物は、発見可能及び識別可能なリソースのコレクションである。ウェブ出版物に関する情報は、§ 2. 第 I 部：出版物のマニフェストで定義されているフォーマットの実装であるところの、マニフェストで表現される。マニフェストは、ユーザーエージェントが、ウェブ出版物の境界及びそのリソース間の接続を理解できるようにするものである。

マニフェストは、構成するリソースを超えたアイデンティティと性質がある出版物であるため、ウェブ出版物を記述するメタデータを包含する。マニフェストは、単一の連続した作品としてリソースを接続する方法であるところの、ウェブ出版物に属する全てのリソースのリスト及びデフォルトの読み上げ順序も提供する。

ウェブ出版物は、どのようなリソースでも、マニフェストへのリンクの存在を介して発見可能である（つまり、HTTP リンク ヘッダー又は HTML link 要素 [html] を使用することによって）。リンクされたマニフェストは、（プライマリ エントリ ページと呼ばれる特別な場合）リンクを含むリソース内に直接かそれとも別の JSON-LD ドキュメントに含まれる。

ウェブ出版物の確立により、ユーザー エージェントは、固有の読書ニーズに合わせて特別に調整された新しい体験を構築できる。

3.2 適合クラス

この仕様は、二つの適合クラスを定義しており、ひとつはウェブ出版物用で、もうひとつはそれらを処理するユーザー エージェント用である。

ウェブ出版物は、次の基準を満たす場合、この仕様に適合する。

  • § 3.4 マニフェストに適合するマニフェストを持ち、
  • § 3.3 ウェブ出版物の構築で定義されている構築要件に忠実である。

ユーザー エージェントは、次の基準を満たす場合、この仕様に適合する。

  • ウェブ出版物の適合マニフェストを処理できる。

3.3 ウェブ出版物の構築

3.3.1 出版物の境界

ウェブ出版物は、そのコンテンツを表現する有限なリソースのセットで構成される。この規模は境界として知られ、そのマニフェスト内で定義される — これは、デフォルトの読み上げ順序とリソース リストに掲載されているリソースの和集合から取得される。

リソースがウェブ出版物の境界内にあるかどうかを判断するために、ユーザー エージェントは、リソースの絶対 URL と和集合から取得したリソースの絶対 URLs と比較しなければならない（MUST）。リソースが一覧で識別される場合、それはウェブ出版物の境界内にある。他の全てのリソースは、ウェブ出版物の外部にある。

ウェブ出版物の境界内のリソースは、同じドメインを共有する必要はない。

3.3.2 リソース

ウェブ出版物は、少なくともひとつの HTML ドキュメント [html]（プライマリ エントリ ページ）を包含しなければならない（MUST）。

そうでなければ、ウェブ出版物は、デフォルトの読み上げ順序と他のリソースの依存の両方で、任意のメディア タイプのリソースを参照してもよい（MAY）。

Note

ウェブ出版物にリソースを追加するとき、ユーザー エージェントのサポートを考慮されたい。必要に応じて、プログレッシブ エンハンスメントの技術を使用とフォールバック コンテンツを提供することで、好みのユーザー エージェントに関係なく、ユーザーに対してより一貫した読書体験を保証するだろう。

3.3.3 プライマリ エントリ ページ

プライマリ エントリ ページは、ウェブ出版物の優先的な開始リソースを表し、マニフェストの発見を可能にする。ウェブ出版物のアドレスにアクセスするときに返されるのは [HTML] リソースで、デフォルトの読み上げ順序又はリソース リストのいずれかに包含されなければならない（MUST）。

任意のリソースはマニフェストにリンクできるとはいえ、通常、プライマリ エントリ ページはウェブ出版物を紹介し、コンテンツへのアクセスを提供する。単一ページのウェブ出版物の場合は、全てのコンテンツを含むことも、複数ドキュメントのウェブ出版物は読み始めるためのナビゲーション支援を提供することもできる。ユーザーの消費しやすさを促進するために、プライマリ エントリ ページは目次を含めるべきである（SHOULD）。

プライマリ エントリ ページはデフォルトの読み上げ順序に包含されている必要はなく、また、包含されているときに最初のドキュメントとして掲載される必要もない。この仕様は、様々なアプローチに対して柔軟性を提供するために、ドキュメントの正確な性質を意図的に仕様を曖昧なままにしている。しかしながら、デフォルトの読み上げ順序が指定されていない場合、ユーザー エージェントはプライマリ エントリ ページを使用して読み上げ順序を作成するだろう。

プライマリ エントリ ページのアドレスは、ウェブ出版物の正規の識別子でもある（つまり、一意の識別子として機能する）。

情報がマニフェストから省略されている特定の場合では、ユーザー エージェントはときどき情報のフォールバック ソースとしてプライマリ エントリ ページを使用するだろう（言語と基本方向及びタイトルを見よ）。

3.3.4 目次

目次は、ウェブ出版物の主要なセクションの構造アウトラインを反映したリンクの階層リストを提供する。

目次は、リソースのひとつである [html] 要素（典型的に、nav 要素）を介して表される。この要素は、role 属性 [html] 値「doc-toc」 [dpub-aria-1.0] によって識別されなければならず（MUST）、その role 値を持つ（ドキュメント ツリー順 [dom] の）ドキュメント内における最初の要素でなければならない（MUST）。

目次がプライマリ エントリ ページに配置されていない場合、マニフェストは構造を含むリソースを識別するべきである（SHOULD）。

例 56 ：プライマリ エントリ ページが目次を包含している場合、マニフェスト内でそれを参照する必要はない 
<head>
    …
<script type="application/ld+json">
    {
"@context"        : ["https://schema.org","https://www.w3.org/ns/wp-context"],
"type"            : "TechArticle",
        …
"id"              : "http://www.w3.org/TR/tabular-data-model/",
"url"             : "http://www.w3.org/TR/2015/REC-tabular-data-model-20151217/",
        …
    }
</script>
    …
</head>
<body>
    …
<section role="doc-toc">
        …
</section>
    …
</body>

指定されたとき、目次は、少なくともひとつのリソースへのリンクを包含しなければならず（MUST）、そして、全てのリンクは出版物の境界内のリソースを参照するべきである（SHOULD）。

目次に含まれるリソースを識別する方法についての情報は、目次プロパティの定義を参照されたい。

3.3.5 ページ リスト

ページ リストは、コンテンツ内の静的なページ分割ポイントへのナビゲーションを提供するリンクのリストである。これらのロケーションは、コンテンツへのアクセスの調整をユーザーに許可するが、ロケーションの正確な性質はコンテンツ作成者が定義する。これらは通常、ウェブ出版物のソースである印刷ドキュメントのページに対応するが、ナビゲーションを容易にするために追加された完全にデジタル作成物である場合がある。

ページ リストは、リソースのひとつである [html] 要素（典型的に、nav 要素）を介して表される。この要素は、role 属性 [html] 値「doc-pagelist」 [dpub-aria-1.0] によって識別されなければならず（MUST）、その role 値を持つ（ドキュメント ツリー順序 [dom]）ドキュメント内における最初の要素でなければならない（MUST）。

ページ リストが、プライマリ エントリ ページに配置されていない場合、マニフェストは構造を含むリソースを識別するべきである（SHOULD）。

指定されているとき、少なくともひとつのリソースへのリンクを包含しなければならない（MUST）ことを除き、ページ リスト自体に要件はない。

ページ リストに含まれるリソースを識別する方法についての情報は、pagelist プロパティの定義を参照されたい。

3.4 マニフェスト

3.4.1 要件

ウェブ出版物マニフェストのプロパティの表現は、ウェブ出版物に固有の二つ（出版物のアドレスと正規の識別子）だけでなく、デジタル出版物に対して一般に定義されているものも組み合わせて包含する。

これらのプロパティの表現のための要件は次のとおりである。

必須：
  • アドレス
  • デフォルトの読み上げ順序
  • タイトル
推奨：
  • アクセシビリティ
  • 基本方向
  • 正規の識別子
  • クリエイター
  • 言語と基本方向
  • リンク
  • 最終変更日
  • 発行日
  • 組み方向
  • リソース リスト
Note

これらのプロパティは、作成されたマニフェストで全てシリアル化する必要はない。マニフェストに必要かそれとも他の情報から正規のマニフェストにコンパイルできるかを判断するために、各プロパティの定義を参照する。

さらに、次のリソースの包含を推奨する（RECOMMENDED）。

  • アクセシビリティ レポート
  • カバー
  • ページ リスト
  • プライバシー ポリシー
  • 目次

3.4.2 プロパティ

3.4.2.1 デフォルトの読み上げ順序

ウェブ出版物の readingOrder プロパティは、次の方法で出版物マニフェスト readingOrder プロパティを拡張する。

  • デフォルトの読み上げ順序は、プライマリ エントリ ページのみで構成されているときは、省略してもよい（MAY）。デフォルトの読み上げ順序が存在しないとき、ユーザー エージェントは、正規のマニフェストをコンパイルするときにプライマリ エントリ ページのためのエントリを包含しなければならない（MUST）。
3.4.2.2 タイトル

ウェブ出版物の title プロパティは、次の方法で出版物マニフェスト title プロパティを拡張する。

  • タイトルが作成されたマニフェストに包含されていない場合、ユーザー エージェントは、正規のマニフェストを生成するとき、ウェブ出版物のプライマリ エントリ ページにある title 要素 [html] の値を使用しなければならない（MUST）。
Note

ウェブ出版物が、（例えば、本の章ごとになど）複数の HTML リソースで構成される場合、title 要素に依存することは、HTML 定義が、リソースのコレクションではなく、同封している HTML ドキュメントの「メタデータ」としてこの要素を定義するので、セマンティクスとして問題がある。一方、この要素の使用は、（例えば、学術雑誌の記事など）単一の HTML ドキュメントで構成されるウェブ出版物の場合に推奨される。

3.5 関連性

3.5.1 マニフェスト

プライマリ エントリ ページは、発見できるようにするためマニフェストへのリンクを提供しなければならない（MUST）。このリンクを提供できる唯一のリソースである。

プライマリ エントリ ページは、マニフェストを埋め込むことができる唯一のリソースでもある。プライマリ エントリ ページにマニフェストを埋め込むことを推奨するが（RECOMMENDED）、マニフェストは外部にあってもよい（MAY）。

Note

埋め込みは、マニフェストが HTML ページに埋め込まれているとき、検索エンジンは、JSON-LD フォーマットの schema.org メタデータでしか処理しない可能性があるため、望ましい任意である。

3.5.2 出版物

プライマリ エントリ ページへのリンクは、ウェブ出版物を成立させている全てのリソースが包含されてもよい（MAY）。

Editor's note

リンクのための使用に対する関係性を決定する必要がある。

ウェブ出版物にリソースをリンクすることは、ユーザー エージェントが、リソースにユーザーが到達する方法に関係なく、ウェブ出版物に属するリソースを確認できるので、できる限り推奨される。

ウェブ出版物のリソースは、マニフェストに直接リンクしてはならない（MUST NOT）。そのようなリンクを提供できるのは、プライマリ エントリ ページのみである。

Note

3.6.1 マニフェストを取得する

プライマリ エントリ ページから開始する、マニフェストを取得する手順は、次のアルゴリズムによって与えられる。

  1. プライマリ エントリ ページに、メディア タイプ text/html 又は application/xhtml+xml がない場合、このアルゴリズムを終了する。
  2. プライマリ エントリ ページのトップレベル ブラウジング コンテキストの Document から、origin は Document の起源になり、manifest link は、publication トークンを含む rel 属性の Document のツリー順で最初の link 要素になる。
  3. origin が、[html] 曖昧な起源の場合、このアルゴリズムを終了する。
  4. manifest link が null の場合、このアルゴリズムを終了する。
  5. manifest link の href 属性値が、空の文字列の場合、このアルゴリズムを終了する。

  6. manifest link の href 属性値が、[URL] origin と等しい場合、

    1. Document の識別子 id を識別する空ではないフラグメントがある場合、

      1. id 属性が id と等しく、type 属性が application/ld+json と等しい embedded manifest script をツリー順の最初の script 要素とする。
      2. embedded manifest script が null の場合、このアルゴリズムを終了する。
      3. embedded manifest script の子のテキスト コンテンツを text とする。
      4. script element の baseURI の値を base とする。
    2. そうでなければ、このアルゴリズムを終了する。
    説明

    このブランチは、マニフェストがプライマリ エントリ ページに埋め込まれているときに使用される。アルゴリズムは、script 要素を見つけ、マニフェスト自身を抽出する。ドキュメントの URL、又は著者が設定した場合、利用可能な base 要素の値は、相対 URL を絶対 URL に変換するために使用される。

  7. そうでなければ、
    1. 要素のベース URL と比較して、href 属性の値を解析した結果を manifest URL とする。解析が失敗した場合、これらの手順を中止する。
    2. URL は manifest URL であり、コンテキストは Document のブラジング コンテキストと同じなので、request を新しい [fetch] リクエストとする。
    3. manifest link の crossOrigin 値が、「use-credentials」の場合、request の認証情報を「include」に設定する。
    4. request でフェッチを実行した結果を待ち、response を結果にする。
    5. response がネットワーク エラーの場合、このアルゴリズムを終了する。
    6. UTF-8 デコードした response 本文の結果を text とする。
    7. manifest URL の値を base とする。
    説明

    このブランチは、マニフェストが別のファイルにある場合に使用される標準操作を実行して、ウェブからマニフェストを取得するために標準的な操作を実行する。マニフェスト ファイルの URL は、相対 URL を絶対 URL に変換するために使用される。

text に空でない文字列が含まれている場合、ベース URL としての base、プライマリ エントリ ページとして Document を伴って、処理段階の最初のステップへの入力とする。そうでなければ、このアルゴリズムを終了する。

B. 機械処理可能な目次

B.1 導入

このセクションは規定ではない。

ページ内及びサイト間のナビゲーションを容易にするために、HTML は、リンクのリストを表現するために nav 要素 [html] を使用する。デフォルトでは本質的に汎用ではあるが、nav 要素の目的は、role 属性 [html] の使用によって、より明確に識別することである。特に、[dpub-aria-1.0] 語彙の doc-toc ロールは、ウェブ出版物の目次として nav 要素を識別する。

識別可能な目次を包含することは、任意のデジタル出版物を生成するためのアクセシブルな方法だが、HTML マークアップの柔軟性のために、（例えば、任意のページから利用可能なカスタム ビューを提供するためなど）リンクの意味のある階層を抽出しようとするユーザー エージェントにとっての課題を提示する。様々な用途の目次が重複するのを避けるために、このセクションは、ユーザーエージェントの抽出に十分な構造を提供しながら、ヒューマン フレンドリーかつ一般的な用途の構文を定義する。

著者は、目次を構築するためのリスト（順序付きまたは順序なし）を選択できる。これらのリスト内の各リンクをアンカー タグ（a 要素）でタグ付けすることにより、ユーザー エージェントは、必要な情報を周囲のコンテンツ（補助）又は追加されているスタイル的なタグとを容易に区別できる。目次は、アクティブなリンク（href 属性を含む）と非アクティブなリンク（href 属性を除く）の両方で構成でき、（例えば、特定の見出しへのリンクを省略したり、 プレビュー内の特定のコンテンツへのリンクなど）目次の構成方法にさらなる柔軟性を提供できる。

B.2 HTML 構造

このセクションは規定ではない。

機械可読な目次は、[html] nav 要素内で定義される。§ 3.3.4 目次で説明されているように、この nav 要素は、ドキュメントの最初の要素であり、値 doc-toc を持つ role 属性によって識別される必要がある。

nav 要素のコンテンツ モデルは、制限がないけれども、ユーザー エージェントは、次のマークアップ ガイドラインに従うときのみ、使用可能な目次を抽出できるだろう。

目次のタイトル

目次のタイトルはオプションであるが、必要なときにユーザー エージェントがプレースホルダー タイトルを生成するのを避けるために、タイトルの追加することを勧める。タイトルは、[html] h1 から h6 要素のいずれかを使用して指定される。最初のそのような要素のみがタイトルとして認識されることに注意されたい。見出し要素がリンクのリストの前に見つからない場合、ユーザー エージェントは、見出し要素が指定されていないと見なす。

リンクのリスト

nav 要素内で遭遇した最初の [html] ol 又は ul リスト要素は、コンテンツへのリンクを定義するリストが含まれていると想定される。例えば、このリストは、アルゴリズムがその処理に関係のない無視する要素のため、たとえ div 要素にネストされていて発見されるだろう。しかしながら、内部コンテンツは評価されないため、リストはスキップする要素の内部に発生することはできない。

nav 要素にこれらの要素のいずれも含まない場合、ユーザー エージェントは（例えば、機械でレンダリングされたオプションは利用できないなど）使用可能な目次を含むものとしてデジタル出版物を登録しない。

ブランチ

目次がリンクのツリーと見なされる場合、リンクのリスト内における各リスト項目（li 要素）は、ひとつのブランチを表現する。これらの各ブランチは、ユーザーに提示するために名前とオプションの目的地を持つ必要があり、この情報は、ネストされていれば、リスト項目内で発見された最初 a 要素から取得される（繰り返しになるが、スキップする要素内の全ての a 要素は除外される。）

ブランチのリンクの目的地は、指定されているとき、a 要素の href 属性から取得される。この属性は、リンクが利用できない（例えば、プレビューなど）又は関連性がない（例えば、グループ化したヘッダーなど）場合、省略できる。コンテンツへのリンクを提供するとき、（rel 属性で）リンクされたドキュメントの関係及び（type 属性で）リンクされたリソースのメディア タイプを指定することも可能である。

ブランチにラベルを付ける a 要素を見つけた後、ユーザー エージェントは別のリスト要素（つまり、サブブランチ）のマークアップを調査を続ける。リストが見つかった場合、ネストされたブランチがもはや処理されなくなるまで、同様にリンクを抽出するために処理する。

スキップする要素

誤った解釈を避けるために目次を解析するとき、要素の小さなセットは無視される。これらは、[html] セクショニング コンテンツ要素とセクショニング ルート要素である。これらが無視される理由は、独自のアウトラインを定義できるためである（つまり、自己完結型であり、必ずしもコンテンツリンクの構造に関連していない埋め込みコンテンツを表すことができる。）

隠された要素はユーザーによって直接アクセスされることを目的としていないため、hidden 属性が設定されている要素もスキップされる。

これらの要素は nav 要素に包含できるけれど、（たとえば、コンテンツへの全てのリンクを含むリスト アイテムの周りを section 要素をラップしないようにするなど）それらの中に重要なコンテンツを埋め込まないように注意する必要がある。

無視する要素

目次の抽出に関連せず、スキップされない全ての要素は、無視される。スキップする要素とは異なり、無視することは、ユーザー エージェントが関連コンテンツを検索し続けることを意味し、使用できるタグ付けに関して柔軟性が向上する。

B.3 ユーザー エージェントの処理

このセクションは、nav 要素から目次を抽出するアルゴリズムを定義する。これは、DOM ツリーのノードを、ツリー順に walk することに関して定義されており、各ノードは、ウォーク中にそのノードが入ったとき及び出たときにアクセスされる。ノードがアクセスされるたびに、それは入力又は終了イベントをトリガーしていると見なすことができる。いくつかの手順では、ユーザーエージェントにコンテンツの処理方法の選択及び様々なプレゼンテーション モデルに柔軟性を提供する。

Note

理解を助ける目的のために、このセクションの例は、JavaScript オブジェクトとして目次の構造を示す。ユーザー エージェントは、適切な言語と形式で結果の構造を処理及び内部化できる。

このアルゴリズムの目的のために、リスト要素は [html] ol 又は ul 要素のいずれかとして定義される。

次のアルゴリズムは、role 属性値 doc-toc を使用して、ドキュメント順に最初の nav 要素をルートとする DOM サブツリーの walk のために適用しなければならない（MUST）。全ての説明は有益である。

  1. toc は目次を表現するオブジェクトとし、次のように初期化する。

    1. 目次のタイトルを表現する toc の name プロパティを作成し、空の文字列を設定する。
    2. 目次の全てのブランチを表現する toc の entries プロパティを作成し、空の配列を設定する。
    説明

    この手順は、目次のタイトルとブランチを保存する toc オブジェクトを初期化する。

    例 65 ：デフォルトの toc オブジェクトの視覚化 
    {
"name": '',
"entries": []
}

  2. スタックの初期化。

    説明

    スタックは、まだ完了していないブランチを保持するために使用される。新しいサブブランチに遭遇すると、親を後で取得できるようにスタックへプッシュする。

  3. 現在の toc ブランチを null を設定した変数にする。

    説明

    現在の toc は、現在処理されている目次のブランチを表現するオブジェクトを保持するために使用される。

  4. 目次の構築元である nav 要素から始め、ツリー順で DOM 上を walk し、walk の入力及び終了するときに、各要素に対して次の最初の関連手順をトリガーする。

    1. 見出しコンテンツ要素を入力するとき。

      次の手順を実行する。

      1. スタックが空で、toc の name プロパティが空の文字列である場合、次のいずれかのように name プロパティを設定する。

        • （HTML タグを保持するための）要素の子孫コンテンツ。
        • （例えば、要素のアクセシブルな名前 [accname-1.1] を計算することによってなど）子孫コンテンツから取得されたテキスト文字列。

        （例えば、プレゼンテーション要素を削除し、全ての先頭と末尾の空白をトリミングした後など）name の結果の値が空の文字列である場合、name プロパティをプレースホルダー値かそれとも null に設定する。

      2. 要素を終了し、次の要素で処理を続行する。
      説明

      この手順は、目次の見出しを識別する。見出しは、toc name プロパティの値が空の文字列の場合にのみ処理される（つまり、見出しはまだ検出されていない）。

      ユーザー エージェントが見出し要素の子孫コンテンツに name を設定するかそれとも、そこからテキスト文字列を生成するかは、（例えば、画像、MathML、ルビ、及び簡単にテキストへ変換できない他のコンテンツを保持するためなど）プレゼンテーションで子孫タグを再利用するかどうかによって異なる。

      例 66：見出しのある toc オブジェクトの視覚化 
      {
"name": "Contents",
"entries": []
}

      name が空の文字列ではない、または null の場合、すでに前の見出しと遭遇しているか、（例えば、見出しはリンクのリストをたどらないため、リストは既に処理されているなど）nav 要素に見出しがないことを示すコンテンツと遭遇している。

      例 67 見出しのない toc オブジェクトの視覚化 
      {
"name": null,
"entries": []
}

      見出しが識別されてない場合、ユーザー エージェントは、後で使用するために独自の見出しを提供できる。

    2. リスト要素を入力するとき

      次の手順を実行する。

      1. toc の name プロパティが空の文字列の場合、name を null に設定する。

      2. 現在の toc ブランチが null でない場合。

        1. 現在の toc ブランチの entries プロパティが null 又は空でない配列の場合、要素を終了し、次の要素で処理を続行する。
        2. そうでなければ、スタックに現在の toc ブランチのオブジェクトをプッシュし、現在の toc ブランチを null に設定する。

      3. そうでなければ、スタックが空の場合。

        1. toc の entries プロパティが null 又は空でない配列の場合、要素を終了し、次の要素で処理を続行する。
        2. そうでなければ。何もしない。
      説明

      このアルゴリズムは、単一のブランチ又は nav 要素のルートで複数のリストを処理せず、そのため、リストと既に遭遇している場合（entries プロパティにひとつ以上のブランチが含まれているか、null に設定されている）、このリストはスキップされる。

      リストと遭遇し、目次（toc）にまだ名前がない場合（つまり、見出し要素と遭遇していない）、目次は見出しがないと想定される（つまり、目次の見出しはエントリの最初のリストの後に表示できない）。name プロパティの値は、これ以上は見出しが適用されないため、空の文字列から null に変更される。

    3. リスト要素を終了するとき

      スタックが空でない場合、スタックからオブジェクトを取り、それに現在の toc ブランチを設定する。

      説明

      これは、全ての子ブランチが処理された後、現在の toc ブランチが親オブジェクトにリセットされる。

    4. リスト項目要素を入力するとき場合

      次の手順を実行する。

      1. 新しいオブジェクトに現在の目次オブジェクトを設定する
      2. オブジェクトの name、url、type、及び rel プロパティを作成し、空の文字列を設定する。
      3. オブジェクトに entries プロパティを作成し、空の配列を設定する。
      説明

      各リスト項目は目次の新しいブランチの候補を表現しているため、新しいブランチに遭遇すると、新しい空のオブジェクトが現在の toc ブランチに作成される。

      例 68 ：新しいブランチ オブジェクトの視覚化 
      {
"name": '',
"url": '',
"type": '',
"rel": '',
"entries": []
}

      このオブジェクトは、子孫の a 要素として情報が投入され、リストに遭遇する。

    5. リスト項目要素を終了するとき

      次の手順を実行する。

      1. 現在の toc ブランチの entries プロパティが空の配列を含む場合、値を null に設定する。

      2. スタックがひとつ以上のエントリを含む場合。

        1. 現在の toc ブランチの entries プロパティに空でない配列が含まれ、その name プロパティが空の文字列である場合、その name をプレースホルダ値又は null に設定する。
        2. 現在の toc ブランチの entries プロパティが空の配列が含まれ、その name プロパティが空の文字列の場合、現在の toc ブランチを null に設定し、この処理手順を終了する。

        スタックの一番上にあるオブジェクトの entries プロパティの配列に現在の toc ブランチを追加する。

      3. そうでなければ、toc の entries 配列に現在の toc ブランチ内のオブジェクトを追加する。

      4. 現在の toc ブランチに null を設定する。

      説明

      リスト項目を終了すると、現在のブランチの処理が完了したことを示す。親の entries 配列にこのブランチを追加する前に、ブランチに名前及び/又はサブブランチがあるかどうかをテストする必要がある。名前はないがサブブランチがある場合、ブランチは保持される。ユーザー エージェントは、独自に作成のプレースホルダー値を提供するか、値を null に設定できる。名前またはブランチがない場合、無効になり破棄される。

      ブランチをマージする場所を決定するために、スタックがチェックされる。スタックにオブジェクトがない場合、ルートの toc オブジェクトの entries プロパティに追加される（つまり、最上位ブランチである）。そうでなければ、スタック内にある直前のオブジェクトの entries プロパティに追加される。

      最終の手順として、現在の toc ブランチは null に戻されてリセットされる。

      例 69：ブランチ マージの視覚化

      次の二つのオブジェクトが、現在の toc ブランチにある場合

      {
"name": "Section 1",
"url": "http://example.com/contents.html#s1",
"type": "text/html",
"rel": null,
"entries": []
},
{
"name": "Section 1.1",
"url": "http://example.com/contents.html#s1.1",
"type": "text/html",
"rel": null,
"entries": null
}

      マージ後、次の単一オブジェクトのみが残る。

      {
"name": "Section 1",
"url": "http://example.com/contents.html#s1",
"type": "text/html",
"rel": null,
"entries": [
      {
"name": "Section 1.1",
"url": "http://example.com/contents.html#s1.1",
"type": "text/html",
"rel": null,
"entries": null
      }
   ]
}

    6. アンカー要素を入力し、現在の toc ブランチが null でない場合。

      次の手順を実行する。

      1. 現在の toc ブランチの name プロパティが、空の文字列でない場合、何もしない。

      2. そうでなければ

        1. 次のいずれかに現在の toc ブランチの name プロパティを設定する。

          • （HTML タグを保持するための）アンカー要素の子孫コンテンツ。
          • （例えば、要素のアクセシブルな名前 [accname-1.1] を計算することによってなど）子孫コンテツから抽出したテキスト文字列。

          （例えば、プレゼンテーション要素を削除し、先頭と末尾の空白を全てトリミングした後など）name の結果の値が空の文字列の場合、name プロパティを null に設定する。

        2. 要素に href 属性があり、属性の URL がデフォルトの読み上げ順序又はリソース リストのリソースで解決される場合、現在の toc ブランチの url プロパティを値に設定する。そうでなければ、プロパティを null に設定する。
        3. 要素に type 属性があり、先頭と末尾の空白をトリミングした後の属性の値が空の文字列でない場合、現在の toc ブランチの type プロパティをその値に設定する。そうでなければ、プロパティを null に設定する。
        4. 要素に rel 属性があり、先頭と末尾の空白をトリミングした後の属性の値が空の文字列でない場合、現在の toc ブランチの rel プロパティをその値に設定する。そうでなければ、プロパティを null に設定する。

        要素を終了し、次の要素で処理を続行する。

      説明

      この手順は、ブランチの name と url プロパティの値を取得するためにアンカータグを処理する。

      現在のブランチの名前がすでに定義されている場合、この要素の処理は終了する（つまり、単一のブランチにおける複数のリンクの処理を回避するため）。

      ユーザー エージェントが a 要素の子孫コンテンツにエントリの name を設定するかそれとも、そこからテキスト文字列を生成するかは、（例えば、画像、MathML、ルビ、及び簡単にテキストへ変換できない他のコンテンツを保持するためなど）プレゼンテーションで子孫タグを再利用するかどうかによって決まる。

      href 属性を識別することに加え、この仕様の要件を満たすために、デジタル出版物に属するリソースを解決する必要がある。そうでない場合、ブランチは保持されるが、エントリはリンクできない。

      リンクのターゲットに関する追加情報（リソースのタイプとその関係）も保持される。

      例 70 ：SVG 画像へのリンクの視覚化 
      {
"name": "In the Beginning",
"url": "http://example.com/page1.svg",
"type": "image/svg",
"rel": null,
"entries": []
},

    7. セクショニング コンテンツ要素、セクショニング ルート要素、又は隠された属性を持つ要素を入力するとき。

      要素を終了し、次の要素で処理を続行する。

      説明

      セクショニング要素とセクショニング ルート要素は独自のアウトラインを定義できるため、それらに由来すると目次の生成に問題が生じる（つまり、直接関連しないコンテンツが含まれる場合がある）。結果として、子コンテンツが処理されることを防ぐために遭遇したとき、スキップされる。

    8. そうでない場合：何もしない。

      説明

      他の全ての要素について、この手順は、子孫要素の処理を継続できる。

  5. DOM walk の完了後、toc の entries プロパティに空でない配列が含まれている場合、toc は機械処理された目次を表現する。

    そうでなければ、デジタル出版物は、機械でレンダリングする目的で使用できる目次を持たない。

    説明

    ルート toc オブジェクトの entries 配列にブランチが含まれていない場合（リストが、nav 要素で発見されないかそれともリストに、適合するリスト項目が含まれていないため）、アルゴリズムは利用可能な目次を作成しない。

D. 双方向テキストの例

このセクションは規定ではない。

このセクションは、必要に応じて、消費者が期待する表示を生成できるように、Unicode フォーマット文字を双方向文字列に適用する方法を解説する。最初の強いヒューリスティックが誤った結果を生成する場合には、先頭にフォーマットする文字を文字列に生成すると、最初の強いヒューリスティックは文字列全体に対して正しい基本方向を生成する。

ラテン文字で始まる右から左への文字列は、U+200F RIGHT-TO-LEFT MARK を先頭に付ける必要がある。

メモリー内の文字の順序：  HTML היא שפתסימון.
誤った表示を伝える：  HTML היא שפת סימון.
フォーマット文字のあるソース コード：  "\u200FHTML היא שפת סימון."
期待する表示を伝える：  HTML היא שפת סימון.

アラビア文字で始まる左から右への文字列は、U+200E LEFT-TO-RIGHT MARK を先頭に付ける必要がある。

メモリー内の文字の順序：  'سلام' is hello in Persian.
誤った表示を伝える：  'سلام' is hello in Persian.
フォーマット文字のあるソース コード：  "\u200E'سلام' is hello in Persian."
期待する表示を伝える：  'سلام' is hello in Persian.

「ウェブ出版物の構造」略図の説明
ウェブ出版物の構造の簡略図。ウェブ出版物は二つの要素に分割される。最初の要素は、実際のコンテンツである（マニフェストに掲載されている全ての実物）。この要素は、CSS と、例えば HTML ドキュメント、オーディオ、及び画像、フォントなどの実際の「モノ」に分割される。実際の「モノ」は、出版物のエントリ ページ及び他の全てのドキュメントを包含する追加のアイテムのサブセットである。二つ目の要素は、マニフェスト（JSON）である。マニフェストは、出版物内の全ての「モノ」のリスト、出版物のメタデータ、及びコンテンツのデフォルトの読み上げ順序で構成される標準的な正規のマニフェストを生成するために使用される。図では、エントリ ページがマニフェストにリンクされている必要があることに注意されたい。（ウェブ出版物の略図に戻る。）
